## 2026-03-11
- For phone forms inside a `ScrollView`, do not rely on static bottom padding to keep low `TextInput` fields visible. If operators type near the bottom of the screen, wire focus-driven keyboard scrolling so the active field moves above the iPhone keyboard with a bit of extra clearance.
- For iPhone keyboard avoidance in React Native forms, do not assume `scrollResponderScrollNativeHandleToKeyboard` alone will visibly move fields on-device. Pair real keyboard insets with measured overlap-based `scrollTo(...)` logic that uses the keyboard frame and current scroll offset.
- For waiting-state visuals on phone UIs, a spinning center icon can read weaker than a standard circular loader. If the user references a classic ring loader, keep the center symbol static and move the motion to orbiting/fading dots around it.
- On dense phone action screens, if the user wants more of the pending state visible without scrolling, remove decorative header branding first and merge identity info onto the title row before shrinking the main action card. Preserve the functional hierarchy while reclaiming vertical space.
- When the user corrects an operational label on the home summary surface, update the visible wording literally and do not keep the old field name for convenience. Match the operator's exact term, such as changing `예선 상태` to `도선 상태`.
- For YT Master call notifications, do not stop at server-side push delivery. Operators can already be inside the app on another screen, and local development may run with `EXPO_PUSH_ENABLED=false`, so keep a targeted in-app SSE fallback that still raises a forced local banner and auto-opens the existing `반장 호출` screen for the relevant device.
- When the user gives a popup motion reference video, do not stop at a functional modal. Match the reference interaction pattern with anchored scale-and-rise motion, a dimmed backdrop, and a card surface that reads closer to a peek/pop quick action than a generic sheet.
- When the user refines a peek/pop editor toward a cleaner feel, remove non-essential labels and helper copy before changing behavior. A translucent anchored card should read like a lightweight inline quick edit, not a small settings dialog.
- For translucent popup cards, do not invent background-like decorative blobs inside the card. If the user expects the real screen to show through, use only the actual underlying UI plus subtle border/shadow, not fake glow shapes that can be mistaken for missing background content.
- When the user wants a ChatGPT/iOS-style quick menu feel, prefer a smaller titleless anchored card with a short spring scale/opacity reveal and minimal travel. Treat it like a compact context menu, not a roomy custom popover.
- When the user asks for a more current elastic pop feel, do not keep a uniform scale animation. Use a short two-phase squash-and-release with separate `scaleX/scaleY` curves so the card briefly compresses before snapping open.
- If a popup animation looks choppy on-device, check whether keyboard autofocus is starting at the same time as the pop. Let the card animate first, then focus the input, and prefer one continuous spring over ultra-short stitched timing phases.
- If the popup close still jitters after the open path is fixed, check the close side separately. Closing can stutter for the same reason if the popup starts fading out while the keyboard is still dismissing; in that case, dismiss the keyboard first and close the popup after `keyboardDidHide` or a short fallback delay.
- If the user explicitly wants the popup to disappear before the keyboard moves, do not treat keyboard-first dismissal as "good enough." Start the popup close animation immediately, then dismiss the keyboard only after the visual close has completed.
- When the user wants true iPhone-native menu feel inside this Expo stack, stop over-tuning a custom modal once it becomes clear the target is system UI. Use built-in iOS primitives that are already supported here (`ActionSheetIOS`, `Alert.prompt`) and be explicit that Home Screen quick actions themselves are system-level UI, not an in-app component you can copy 1:1.
- If the user specifically wants the menu to open from the trigger's physical location, `ActionSheetIOS` is not enough. On supported iPhone runtimes, switch to an anchored native context menu (`@expo/ui` `ContextMenu` here) and keep Expo Go on a documented fallback because the native menu module is unavailable there.
- If the user later rejects that native-menu detour and explicitly asks to return to the previous custom interaction, roll back the native branch completely instead of trying to hybridize both. Keep the proven animation-order fix that solved the keyboard contention, but remove the extra runtime/package complexity.
- When operators want an urgent accident path inside YT Driver, do not bury it as just another selectable reason chip. Keep normal reasons in the selection panel and expose the emergency flow as a separate red direct-call control that submits immediately and stays visually distinct in both driver and master views.
- If the user wants an emergency CTA to read more immediately, remove extra helper copy and decorative icon frames before changing layout. Let the siren glyph stand on its own and put the emphasis on one larger direct label, not a two-line explanatory block.
- If the user says a single emergency CTA still feels undersized, do not add new decoration. First increase the card's vertical mass, enlarge the core icon/text, and center the content more tightly so the control reads as a primary urgent action.
- If a row-style emergency CTA still leaves dead space on one side, stop stretching the text block. Center the icon-and-label group as one unit inside the card instead of letting a `flex: 1` body pin the label off-center.
- If the user explicitly says the label itself must sit on the card's exact centerline, do not rely on row balancing. Pin the icon separately on the left and center the text against the full card width so the title stays geometrically centered.
- If the user then calls out a slight remaining imbalance, do not rebuild the card again. Keep the anchored structure and apply a tiny text-side offset for visual centering against the icon's optical weight.
- When a second top-level YT call reason also needs detailed subreasons, do not fork a parallel ad-hoc payload. Expand the shared `reasonDetailCode/Label` model to support both reason families, validate each family explicitly, and reuse the same long-press grouped-picker pattern in the mobile UI.
- If a dense reason-chip picker is just barely overflowing, shorten repeated operator words in the shared labels first (`캐빈`, `노매너`, redundant position words like `1단`) before touching chip font, padding, or component styling.
- If one remaining chip is still causing the last wrap, shorten that single label aggressively to the operator's core noun phrase before touching any layout values. Prefer a direct label like `컨테이너 돌출` over preserving full location wording when the user explicitly prioritizes same-screen fit.
- When operators want ETA wording unified, do not preserve special relative-day phrases like `내일로` or `모레`. Keep `crossedDate` in the payload for logic, but normalize the visible message to the same `종전보다 ...` sentence for both same-day and cross-day changes.
- If the user still sees the old ETA wording after the formatter change, assume a persisted-message reuse path before assuming the formatter failed. Normalize stored alert/adjustment `humanMessage` values again at render/decorate time from `deltaMinutes`, so legacy `어제로`/`내일로` text does not survive in live UI.
- For driver-side pre-call helper summaries, do not leave the selected-detail local state alive after a successful action. If the operator has already sent or cancelled the call, clear the local detail selection so stale gray helper text does not linger between the reason chips and the main call card.

## 2026-03-02

- For Expo monorepo runtime validation, do not rely on `expo-doctor` alone; verify with an actual Metro bundle (`expo export` or live bundle request) because Babel/transitive module gaps can still pass doctor checks.
- In this repo, if iPhone red screen shows missing Babel modules from `expo-router/entry.js`, fix mobile Babel stack first (`@babel/core`, `babel-preset-expo`, required plugins) and confirm `expo export --platform ios` succeeds before retesting Expo Go.
